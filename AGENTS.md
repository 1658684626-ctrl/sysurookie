# EasyEvent / 易赛通 Agent Rules

## 1. 项目简介
EasyEvent（易赛通）是一个面向中小型体育赛事的一站式数字化管理系统课程 MVP。首个演示案例是“珠海城市鲁宾逊趣味定向赛”，但系统必须通过赛事模板配置复用到校园 5km 跑步赛、亲子运动会、篮球赛、企业团建运动赛、徒步/越野类活动等场景。

## 2. 产品定位
EasyEvent 的核心不是普通报名表，而是“赛事配置 + 报名审核 + 签到检录 + 现场执行 + 数据复盘”的一站式运营闭环。系统通过赛事模板配置，把报名组队、资料审核、签到检录、点位打卡、通知确认和数据复盘整合到统一平台，减少微信群、问卷、Excel、纸质签到表和人工核对带来的运营混乱。

## 3. 目标用户
- 赛事组织方：负责创建赛事、配置规则、审核资料、管理签到、发布通知、查看现场状态和赛后数据。
- 参赛者/队伍负责人：负责查看赛事、报名组队、填写成员资料、查看审核与签到状态、完成点位打卡、确认通知。
- 课堂演示观众：需要快速看懂中小型赛事从报名到复盘的完整数字化流程。

## 4. 核心痛点
- 报名、组队、资料收集分散在问卷、表格和聊天群里，容易遗漏。
- 人数规则、材料规则、路线规则依赖人工核对，错误率高。
- 现场签到、出发、点位打卡和异常情况缺少统一状态视图。
- 通知公告靠群消息转发，组织方难以确认谁已读。
- 赛后复盘需要重新整理 Excel，缺少可演示的数据闭环。

## 5. MVP 功能范围
重点展示以下闭环：
- 创建/切换赛事模板。
- 参赛者报名/组队。
- 成员资料填写。
- 系统根据赛事规则判断是否可提交审核。
- 管理端审核通过/驳回。
- 报名成功后生成签到码。
- 现场签到检录。
- 队伍在赛中进行点位打卡或任务提交。
- 管理端查看现场执行状态。
- 管理端发布通知公告。
- 参赛者确认已读。
- 管理端查看数据看板与赛后复盘。

## 6. 不做什么
本项目不要实现：
- 真实支付。
- 真实登录。
- 真实后端。
- 真实数据库。
- 真实短信。
- 真实微信接口。
- 真实 GPS。
- 真实地图导航。
- 真实人脸识别。
- 复杂成绩计时。
- 复杂权限系统。
- 复杂票务系统。
- 复杂志愿者排班系统。

## 7. 技术栈要求
推荐技术栈：
- Vite。
- React。
- TypeScript。
- Tailwind CSS。
- lucide-react 图标。
- clsx 用于条件 className。
- zod 用于前端规则与表单数据校验。
- mock data + localStorage。

可选组件策略：
- 优先使用轻量自定义组件。
- 如引入 shadcn/ui，必须保持范围克制，只添加当前页面真正需要的组件。

禁止在 MVP 阶段引入：
- 数据库、ORM、认证服务、支付 SDK、短信/微信服务、地图/GPS SDK、大型后台框架。

## 8. 代码结构约定
建议后续创建前端项目后使用如下结构：
- `src/app/`：应用入口、路由或页面编排。
- `src/components/`：可复用 UI 组件。
- `src/features/events/`：赛事配置、赛事模板和赛事切换。
- `src/features/registration/`：报名、组队、成员资料、报名状态。
- `src/features/review/`：规则核验和管理端审核。
- `src/features/checkin/`：签到码、签到检录、出发状态。
- `src/features/race/`：点位打卡、任务提交、现场执行状态。
- `src/features/announcements/`：通知公告、已读确认。
- `src/features/analytics/`：数据看板和赛后复盘。
- `src/data/`：mock 数据、模板示例、localStorage 读写。
- `src/types/`：共享 TypeScript 类型。
- `src/lib/`：通用工具函数、状态转换、校验 helpers。

不要把业务规则散落在组件里；状态流、规则校验和模板字段应尽量沉到 `types`、`data`、`lib` 或 feature 内部的纯函数中。

## 9. 中文 UI 要求
- UI 文案默认使用简体中文。
- 产品名称使用“EasyEvent / 易赛通”。
- 页面文字应面向课程演示，清楚表达当前流程和状态。
- 不要在界面中堆砌开发说明、键盘快捷键或技术实现解释。
- 管理端偏工作台风格，信息密度适中，便于扫描和对比。

## 10. mock data / localStorage 要求
- MVP 使用 mock data 初始化演示数据。
- 可用 localStorage 保存报名、审核、签到、打卡、通知确认等演示状态。
- localStorage 数据结构必须容易重置，便于课堂演示恢复初始状态。
- 不要伪装成真实后端；相关代码和 README 应明确说明数据仅用于课程 MVP。

## 11. 赛事模板配置要求
所有赛事都应该通过 `EventConfig` 配置，不要把珠海赛事写死在组件里。赛事模板至少支持：
- 赛事名称。
- 赛事时间。
- 赛事地点。
- 报名模式：个人报名/队伍报名。
- 项目/路线。
- 人数规则。
- 成员字段。
- 材料规则。
- 审核规则。
- 签到方式。
- 点位列表。
- 通知类型。
- 数据复盘指标。

珠海城市鲁宾逊趣味定向赛只应作为一个 `EventConfig` 示例。

## 12. 报名状态流要求
报名状态建议使用：
`草稿中 -> 待完善 -> 待审核 -> 审核驳回 / 审核通过 -> 报名成功`

规则：
- 资料不完整时不能进入待审核。
- 系统应能解释未满足的人数、字段或材料规则。
- 审核通过后才生成或展示签到码。

## 13. 审核状态要求
审核状态应至少区分：
- 未提交。
- 待审核。
- 审核通过。
- 审核驳回。

管理端审核必须能看到报名资料、规则核验结果和驳回原因。驳回后参赛者端应能回到资料补充流程。

## 14. 签到状态要求
签到状态建议使用：
`未签到 -> 已签到 -> 已出发`

规则：
- 只有报名成功的个人/队伍才能签到。
- 签到码是 MVP 演示码，不接入真实扫码硬件。
- 已签到和已出发应在管理端现场视图中可见。

## 15. 点位打卡状态要求
点位状态建议使用：
`未到达 -> 已到达 -> 任务已提交 -> 任务通过`

赛中执行状态建议使用：
`未出发 -> 进行中 -> 异常关注 -> 已完赛`

规则：
- 点位列表来自 `EventConfig`。
- 任务提交使用 mock 文本/图片占位，不接入真实上传服务。
- 异常关注用于演示迟到、缺资料、点位未完成等现场问题。

## 16. 通知公告要求
通知公告状态建议使用：
`未发布 -> 已发布 -> 部分确认 -> 全部确认`

规则：
- 管理端可创建和发布演示通知。
- 参赛者端可确认已读。
- 管理端可查看确认人数和未确认对象。
- 不接入真实短信、微信、邮件或推送服务。

## 17. 数据复盘要求
数据复盘应围绕课堂演示展示：
- 报名人数/队伍数。
- 审核通过率和驳回原因分布。
- 签到率、出发率、完赛率。
- 点位完成率。
- 通知确认率。
- 异常关注数量。

数据看板应使用 mock/localStorage 数据计算，不依赖后端。

## 18. 每次修改后需要运行的检查命令
后续创建前端项目后，每次修改应根据项目实际脚本运行：
- `npm run lint`
- `npm run typecheck`
- `npm run test`（如已配置）
- `npm run build`

如果项目使用 pnpm/yarn，应替换为对应命令。若某个脚本尚未配置，需在最终说明里明确“未运行原因”。

## 19. 课堂演示优先级
课堂展示优先保证完整闭环，不追求真实上线复杂度。优先级从高到低：
1. 可配置赛事模板。
2. 报名/组队。
3. 成员资料填写。
4. 资料审核/规则核验。
5. 管理端审核。
6. 签到检录。
7. 点位打卡。
8. 现场通知与确认。
9. 数据看板。
10. 赛后复盘。
11. 演示数据。
12. README 说明。

后续开发中，所有功能选择都应服务于“中小型体育赛事运营闭环”的课堂可演示性。
